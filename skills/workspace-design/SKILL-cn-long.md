# 工作空间设计 Skill

## 目的

把传统“模块 + 过滤器 + 数据表格 + 行内按钮”的管理后台页面，
重构为 **围绕业务动作推进、进度处理、异常应对与决策支持的工作空间页面**。

这个 skill 用于两类任务：

1. **评审现有页面**
   - 判断页面是否仍然有明显管理后台味道
   - 找出哪些地方没有真正服务业务动作
   - 提出工作空间方向的重构方案

2. **从需求直接生成工作空间**
   - 从用户角色、产品目的、业务目标、任务、风险、对象与工作节奏出发
   - 输出页面定位、布局、势态卡、左侧工作队列、两级过滤器、快捷动作、主工作区与上下文抽屉

---

## 核心原则

工作空间不是“把数据列出来供人筛选”，而是：

> **先告诉用户当前局势，再告诉用户该优先处理什么，然后让用户在最合适的视图里直接行动。**

页面应遵循以下优先顺序：

1. **先有页面定位**
2. **再有势态卡**
3. **再有工作队列**
4. **再有过滤器与快捷动作**
5. **最后才是数据视图**

不要反过来从表格和过滤器开始设计。

---

## 固定骨架

默认采用以下骨架：

```text
┌────────────────────────────────────────────────────────────┐
│ 页面标题 + 一句话定位 + 全局快捷动作                         │
├────────────────────────────────────────────────────────────┤
│ 势态卡横条（可横向滚动）                                     │
│ [卡1] [卡2] [卡3] [卡4] [卡5] [卡6]                         │
├────────────────────────────────────────────────────────────┤
│ 页面级过滤器（仅放真正定义整页上下文的条件）                  │
│ [时间] [组织/负责人] [范围] [搜索当前页对象]                  │
├────────────────┬───────────────────────────────────────────┤
│ 左侧工作队列    │ 主工作区                                   │
│                │ 主列本地过滤器 + 快捷动作                    │
│ [ALL]          │                                           │
│ 待处理         │ 视图：卡片 / 列表 / 时间轴 / 看板 / 日历 /   │
│ 风险           │ 详情 / 混合                                 │
│ 待跟进         │                                           │
│                │                                           │
├────────────────────────────────────────────────────────────┤
│ 右侧抽屉 / 侧边面板（按需打开）                               │
└────────────────────────────────────────────────────────────┘
```

默认使用 **两层左侧队列** 与 **两级过滤器**：

- 左侧不是导航，而是工作队列。
- 第 1 层是 `ALL + 分类入口`。
- 第 2 层是 `对象列表 + 返回`。
- 只有当页面级过滤器真的定义了整个页面上下文时，它才应该横跨整个工作空间。
- 主列本地过滤器应放在主工作区内部，只改变当前视图、分组、排序或局部切片方式。

## 什么样的页面才算合格的工作空间

一个合格的工作空间，应同时满足以下要求。

### 1. 页面标题描述的是动作，不是模块

好例子：
- 处理事故
- 审核提交
- 发布版本
- 策划活动
- 运行支持队列
- 发布内容

差例子：
- 管理事故
- 用户列表
- 内容记录
- 任务表

---

### 2. 页面需要一句定位语

格式：

> 这个工作空间帮助【谁】达成【什么目标】，并快速响应【哪些风险或异常】。

例如：

> 这个工作空间帮助发布经理推进今天的部署，并在影响客户之前处理失败检查、阻塞项与回滚风险。

---

### 3. 顶部区域必须先建立“当前局势”

势态卡不是装饰性的 KPI，而是帮助用户判断“现在最重要的是什么”。

它要回答的是“现在发生了什么”，而不是“数据库里一共有多少条记录”。

适合放在势态卡里的内容：
- 今日到期或今日应处理的工作
- 即将触发超时或违约的项目
- 高风险对象
- 本周完成率
- 错误或失败聚类
- 当前流程中的瓶颈
- 产能压力或闲置容量
- 即将到期的访问权限、内容、订阅、审批或任务

不适合放在势态卡里的内容：
- 没有动作价值的静态总量
- 与本页工作无关的全局指标
- 过于密集的报表型统计

势态卡数量要克制。大多数情况下，**4 到 6 张** 是最稳定的范围。再多就容易从工作空间变成 BI 仪表盘。

每张卡最好至少带有以下之一：
- 风险等级
- 趋势
- 截止时间或紧迫度
- 点击后进入对应队列或结果集

势态卡必须能下钻到真实工作：
- 对应队列入口
- 对应风险池
- 对应主工作区状态

卡片应展示摘要与趋势，而不是细节。适合的卡片内容包括：
- 一个主数字
- 上升或下降变化
- 风险标识
- 一句简短说明

势态卡应优先体现 **当前运营周期**，例如今天、本周、本月、当前批次或当前班次，而不是默认给全历史累计值。

一句话总结：**势态卡负责告诉用户，当前哪些局势最值得优先关注。**

---

### 4. 左侧栏必须是两层工作队列，而不是导航树

左侧区域要回答：

> “我应该先处理什么？现在应该打开哪个对象？”

左侧的作用不是“帮用户找东西”，而是“定义当前工作”。它应表达流程、优先级与下一动作。

它不是导航，也不是普通列表，而是系统预先判断后的业务工作清单，把重要工作主动推到前面。

#### 左侧规则：只做两层

默认采用 **严格两层结构**：

##### 第 1 层：ALL + 分类入口

这一层是进入队列的入口层，不是模块树。

典型入口：
- ALL
- 待处理
- 风险
- 待跟进
- 今日重点
- 等待客户
- 升级事项

作用：
- 让用户决定是查看当前全部工作池，还是直接跳到某个有业务意义的队列类别
- 保持队列有组织，但不把页面做成文件夹树

##### 第 2 层：对象列表 + 返回

当用户进入 ALL 或某个分类后，左侧必须切换为可点击的 **对象列表**，并提供明确的 **返回** 操作。

典型对象：
- 09:00 启蒙A班 · 待点名
- 合同 C-2031 · 待付款
- 会员王小明 · 7天内到期
- 音乐B班 · 满班率过低

作用：
- 让用户始终聚焦在真实业务对象上，而不是抽象标签
- 让主工作区立刻进入可处理状态

正确模式是：
- 第 1 层：选一个队列入口
- 第 2 层：选一个真实对象
- 中间：处理这个对象

错误模式是：
- 左侧是模块 / 文件夹 / 标签树
- 中间是记录列表
- 右侧再开详情抽屉

#### 队列分组的设计规则

队列分组必须从业务流程与产品目的出发，而不是先套通用 UI 标签。

它们可以从这些维度推导：
- 流程阶段
- 异常类型
- 时间窗口
- 审批或交接步骤
- 恢复或升级路径
- 责任归属拆分

常见示例分组：
- 待处理
- 风险
- 待跟进
- 今日重点
- 临近截止
- 需要恢复处理的异常

更强的“业务塑形”分组示例：
- 待审核 / 已阻塞 / 已通过 / 已打回
- 校验失败 / 需要重试 / 可发布
- 新请求 / 等待客户 / 升级事项
- 撰写中 / 等待输入 / 待发布
- 未分配 / 处理中 / 有风险 / 已逾期

典型优先顺序通常类似：
- 待处理
- 风险
- 已逾期
- 等待确认
- 待跟进

每个对象项通常应包含：
- 标题
- 对象名、账号、文档、工单、资产或实体名称
- 状态标签
- 时间压力
- 下一动作

左侧栏应保持简短且有意义。它应包含：
- 少量第 1 层队列入口
- 聚焦的第 2 层对象列表

它不应该变成：
- 模块菜单
- 文件夹树
- 第二张大表
- 一堆过滤器堆叠

点击队列入口后，应出现具体对象列表。点击对象后，主工作区应立即进入可处理状态，而不是再把用户送去另一个搜索或筛选页面。

数量是有帮助的，但只是次要信息。真正要传达的不是“有多少”，而是“哪类工作应先做、我现在该打开哪个对象”。

尽量使用业务语言，而不是系统语言。更好的标签例如：
- 待审核
- 待付款
- 即将到期
- 缺少日志
- 风险班级

避免使用这类标签：
- status = pending
- priority = high

一句话总结：**左侧栏是一个两层工作队列：先给 ALL 与分类入口，再给对象列表与返回。**

### 5. 主工作区必须采用最适合业务动作的视图

不要默认用表格。

主工作区的存在是为了支持页面最核心的工作方式，而不是把列表放大。

先问两个问题：
- 用户是在推进一个对象？
- 还是在筛查与处理一批对象？

按工作性质选择视图：

- **卡片视图**：适合扫描与比较
- **列表视图**：适合高密度明细与批量处理
- **时间轴**：适合事件历史、进度、审计与跟进
- **看板**：适合阶段流转、管道、状态推进
- **日历 / 排期视图**：适合时间绑定的规划与资源分配
- **详情视图**：适合对单个对象做深度处理
- **混合视图**：队列 + 详情，或卡片 + 抽屉，或列表 + 预览
- **表单流**：适合多步骤引导式完成动作
- **图表 + 风险列表**：适合监控与异常处理混合场景

规则：
- 视图必须服务于用户动作
- 不要因为后台工具通常都用表格，就机械地继续用表格

列表适合批量处理、比较与复核。时间规划、现场执行、单对象推进，通常不应默认用列表。

主工作区应让用户在当前上下文中完成核心动作，而不是只能“看详情”，还要能“把工作往前推进”。

主工作区应始终围绕一个焦点对象或焦点任务，例如：
- 当前订单
- 当前案例
- 当前文档
- 当前会员
- 当前风险池

不要让主工作区承担所有辅助信息。次级信息、规则说明、候选项与关联对象，可以放进抽屉。

尽量减少页面跳转。理想工作空间支持这样的流：

**进入 -> 判断 -> 执行 -> 完成**

一句话总结：**主工作区是用户真正完成核心工作的地方。**

### 6. 过滤器必须分为两层：页面级过滤器与主列本地过滤器

过滤器的作用是帮助用户快速聚焦，但它不应该喧宾夺主。

过滤器是用来缩小当前范围的，不是用来定义“现在该做什么”。它回答的是“哪些项目满足这些条件”，而不是“我下一步要做什么”。

一个工作空间通常应把过滤器分成 **两层**。

#### 过滤层 1：页面级过滤器

这一层出现在三列结构之上，只在它真的定义了整个页面上下文时才使用。

它回答：

> “这个工作空间当前整体运行在哪个上下文里？”

例如：
- 时间范围
- 站点 / 区域 / 校区 / 工作空间
- 负责人 / 老师 / 顾问 / 处理人
- 学期、项目范围、产品域
- 仅搜索当前页面领域的搜索框

规则：
- 只有当某个条件会同时影响左侧队列和主工作区时，才放在这里
- 保持数量少且稳定
- 大多数情况下，默认只显示 **3 到 4 个页面级过滤器**
- 如果一个过滤条件会直接改变工作队列本身，它大概率属于这一层

#### 过滤层 2：主列本地过滤器

这一层放在主工作区内部，靠近它所控制的视图。

它回答：

> “在当前页面上下文里，我希望现在怎样看这批工作？”

例如：
- 卡片 / 列表 / 看板 / 时间轴视图切换
- 排序方式
- 按状态 / 负责人 / 风险分组
- 只看高风险
- 只看未完成
- 密度或显示模式
- 只影响中列的局部状态切片

规则：
- 这一层只影响主工作区
- 它不应悄悄重建左侧对象池
- 它可以比页面级过滤器更灵活，也更贴近具体视图

#### 怎样判断一个过滤器属于哪一层

用下面的判断方法：

- 如果它会改变 **左侧栏里出现哪些对象**，它就是 **页面级过滤器**
- 如果它只改变 **主工作区如何呈现当前对象**，它就是 **主列本地过滤器**

#### 过滤器的一般规则

常见页面级过滤器：
- 时间范围
- 团队、项目、区域、产品域、工作空间或负责人
- 只有当状态定义了整个页面上下文时，状态才放在这里
- 关键字搜索

常见本地过滤器与快捷动作：
- 视图模式
- 排序
- 分组
- 只看紧急 / 只看未完成 / 只看高风险
- 新建
- 批量更新
- 导出
- 通知
- 分配
- 批准
- 重试
- 标记完成
- 升级处理

要求：
- 不要把本地视图控件横向铺满整个工作空间
- 不要假装所有过滤器都是全局过滤器
- 常用过滤器前置
- 高级过滤器折叠
- 快捷动作靠近主列过滤器，不要散落在每一行按钮里
- 过滤器是用来细化队列或当前视图的，不是用来替代按流程塑形的任务类别

默认只显示高频、稳定、广泛可理解的过滤器。更复杂的条件放进“更多筛选”或高级筛选抽屉。

不要把业务任务伪装成过滤器。像下面这些：
- 待处理
- 已逾期且未完成
- 高风险

通常更适合放在队列入口层，而不是页面顶部过滤器。

过滤器要与主工作区说同一种语言。如果主区是排期视图，就优先给时间、资源与地点；如果主区是风险看板，就优先给风险等级、状态与负责人。

任务推进型页面应使用轻量页面级过滤器；查询分析型页面可以支持更强一些的页面级过滤器。

一句话总结：**页面级过滤器定义整个工作空间的上下文，主列本地过滤器控制当前视图的呈现方式。**

### 7. 使用右侧抽屉承载聚焦的上下文内处理

右侧抽屉用于临时深入，但不把用户带离当前上下文。

它不只是详情面板。它的存在，是为了在不打断主流程的前提下，支持次级决策与补充处理。

适合放进抽屉的内容：
- 快速详情查看
- 状态更新
- 备注或跟进记录
- 历史与时间轴回顾
- 一组紧凑的关联动作
- 辅助上下文，例如责任人、依赖、附件、指标或备注
- 规则解释
- 候选方案
- 关联对象
- 审批意见

原则：
- 不打断主任务流
- 让用户尽量在原页面完成更多工作
- 不要因为查看细节就强制整页跳转

抽屉最适合承载辅助信息与次级动作，而不是主流程。典型例子包括：
- 在几个方案中做选择
- 理解某个状态背后的原因
- 调整参数
- 查看关联数据

如果一打开抽屉，里面其实是一整套复杂页面，那这部分内容更可能应放在主工作区或独立子页面。

抽屉内容应保持轻量、临时且易于关闭。适合承载：
- 详情预览
- 风险解释
- 规则计算
- 候选时间段
- 附加配置

不要把抽屉当成隐藏内容的垃圾桶。抽屉里的每个模块都应回答：
- 它是在支持谁的决策？
- 它为什么不值得长期占据主工作区空间？

一句话总结：**抽屉负责在不打断主流程的前提下，补足决策支持与次级动作。**

## 一般规则

工作空间各区域应有清晰分工：

| 区域 | 它回答的问题 | 核心职责 |
| --- | --- | --- |
| 势态卡 | 现在发生了什么？ | 建立当前局势 |
| 左栏第 1 层 | 我该打开哪个工作池？ | 提供 ALL + 分类入口 |
| 左栏第 2 层 | 我现在该打开哪个对象？ | 提供对象列表 + 返回 |
| 页面级过滤器 | 我当前处于哪个整体上下文？ | 定义整页范围 |
| 主列本地过滤器 | 我想怎样看当前这批工作？ | 细化中列视图 |
| 主工作区 | 我该如何在这里处理它？ | 完成核心工作 |
| 抽屉 | 我还需要哪些辅助信息？ | 支持决策与次级动作 |

快速判断公式：

- 如果一个页面主要是 `过滤器 + 列表 + 详情抽屉`，它更接近 **管理后台**。
- 如果一个页面先用势态卡建立局势，再暴露待办与风险入口，让主工作区直接执行动作，并用抽屉做支持，它更接近 **工作空间**。

最小设计原则：

1. **先看局势，再看全量数据**
2. **先给任务，再让用户找问题**
3. **先建立上下文，再堆过滤器**
4. **让主工作区可执行，而不只是可查看**
5. **让抽屉辅助流程，而不是抢走流程**

## 反模式

如果你看到以下模式，说明这个页面仍然更像管理后台。

### 1. 大表格是页面的重心
- 用户进入页面首先看到一张大表格
- 大部分动作都挂在每行末尾按钮上
- 用户必须先筛选，再自己猜什么值得处理

### 2. 页面定位仍然是“管理某类数据”
例如：
- 管理学员
- 管理课程
- 管理合同
- 管理记录

这说明它是数据管理页，而不是工作空间。

### 3. 过滤器比工作本身更显眼
- 页面上半部分都是筛选控件
- 用户必须先配置查询才能开始工作
- 系统没有主动告诉用户哪里有风险

### 4. 有统计卡，但没有任务承接
- 顶部是一排数字
- 下面仍然是老式列表
- 卡片并没有引导用户进入真实工作

### 5. 所有对象都用同一种视图
- 事故、版本、请求、人员、内容、异常全部都用同一张表
- 设计忽略了对象本身的流程形态

### 6. 把导航伪装成工作空间
- 左侧是模块树
- 中间只是当前模块内容
- 页面没有清晰的任务推进路径

## 设计步骤

当用户给你一个模块、流程或页面需求时，按下面步骤输出。

### 第 1 步：抽取运行上下文
你必须先识别：
- 用户是谁
- 页面最核心的业务目标是什么
- 最常发生的动作是什么
- 哪些风险、失败或异常最关键
- 核心对象是什么
- 节奏是按天、按周、按周期、按截止时间，还是按阶段推进

---

### 第 2 步：定义页面定位
输出：
- 页面标题
- 一句话定位
- 这个页面支撑的核心目标
- 这个页面重点帮助处理的风险或异常

---

### 第 3 步：设计势态卡
最好输出 4 到 6 张势态卡。每张卡都要说明：
- 名称
- 含义
- 为什么它在这个页面重要
- 点击后打开哪个队列、视图或结果集
- 它代表哪个运营周期，例如今天、本周、本月或当前批次

---

### 第 4 步：设计左侧工作队列
从业务流程、工作节奏与产品目的出发定义队列入口。

不要先选一些通用标签，再强行把流程塞进去。入口必须反映工作是如何推进、卡住、升级与完成的。

可用的推导方式：
- 按阶段：受理、审核、批准、执行、核验
- 按风险：阻塞、失败、逾期、升级、异常个案
- 按时间：今天、即将到来、已逾期、等待过久
- 按归属：我的、团队队列、等待外部输入
- 按恢复路径：需要重试、需要修正、需要决策

建议示例：
- 待处理
- 风险
- 待跟进

只有当流程确实不需要更具体的业务标签时，才使用这些通用示例。

队列应保持简短、预过滤，不要变成第二份全量数据集。

为每个入口给出：
- 为什么这个入口存在
- 什么样的工作会进入它
- 第 2 层对象列表长什么样
- 点击对象后主工作区会进入什么处理状态

---

### 第 5 步：设计两层过滤器与快捷动作
明确区分：
- 页面级过滤器
- 主列本地过滤器
- 高级过滤器
- 页面级快捷动作
- 批量动作

还要解释：
- 哪些过滤器会改变整个工作空间上下文，因此应影响左侧队列
- 哪些过滤器只会改变主工作区视图
- 过滤器是否是唯一的上下文机制，还是会由默认场景、搜索或工作队列共同承担一部分
- 默认展示哪 3 到 4 个页面级过滤器
- 默认在主工作区展示哪些本地过滤器
- 为什么额外过滤条件应隐藏在高级筛选后面

---

### 第 6 步：选择主工作区视图
必须明确说明为什么选择：
- 卡片、列表、时间轴、看板、日历、详情，或混合布局

还要说明：
- 主工作区需要承载哪些动作
- 是否支持多视图切换
- 默认视图是什么
- 焦点对象或焦点任务是什么
- 哪些核心动作可以不离开页面直接完成

---

### 第 7 步：设计右侧抽屉
列出：
- 什么会打开它
- 它包含哪些信息
- 哪些动作可以直接在里面完成
- 为什么这些内容适合放在抽屉，而不是主工作区

---

### 第 8 步：给出页面结构草图
输出接近下面的结构：

```text
页面标题 + 一句话定位 + 全局快捷动作
势态卡横条
页面级过滤器
左侧：工作队列（ALL + 分类入口 → 对象列表 + 返回）
中间：主列本地过滤器 + 快捷动作 + 主工作区
右侧：抽屉 / 侧边面板
```

## 评审模式

当用户提供现有页面、截图、HTML 或 PRD 时：

1. 先判断当前设计更像：
   - 工作空间
   - 还是管理后台

2. 再从以下维度逐条评审：
   - 页面定位是否面向业务目标
   - 是否有势态卡
   - 是否由任务与风险驱动
   - 队列分类是否反映业务流程与产品目的
   - 过滤器是否过重
   - 主视图类型是否适合这项工作
   - 是否支持快速处理
   - 详情处理是否保持在上下文中
   - 关键区域是否有清晰的职责分工

3. 最后输出：
   - 主要问题
   - 重构原则
   - 重构后的工作空间方案

## 硬规则

无论是生成还是评审，都必须遵守以下规则：

1. **不要从表格开始设计页面**
2. **不要使用“管理 X”这类标题**
3. **不要让过滤器成为页面主角**
4. **不要把统计卡做成与任务脱节的装饰**
5. **不要默认每一种对象都该用列表**
6. **不要让用户为了详情不断离开当前页面**
7. **始终体现工作节奏、风险与下一动作**
8. **页面一进入就必须回答：现在发生了什么，我该先做什么？**
9. **任务分类必须从业务流程与产品目的出发，不能只套通用标签**
10. **各区域必须职责分明：势态卡负责局势，左栏负责优先处理池与对象入口，过滤器负责范围与视图，主工作区负责动作，抽屉负责支持**
11. **左栏默认应落在 ALL 对象列表，分类入口只是进入对象列表的前置层，不应成为停留终点**
12. **队列入口不是过滤器；会改变优先处理池的入口应留在左栏，不应伪装成顶部筛选项**
13. **页面名、队列名、势态卡名应优先使用业务动作和业务对象语言，而不是系统字段名或泛化管理名**

## 完成标准

只有当结果同时满足以下条件，才算完成：

- 页面名称描述的是动作，而不是数据模块
- 顶部有势态卡，且它们与真实工作联动
- 左侧栏是两层工作队列，而不是菜单树
- 队列入口来源于业务流程，而不是泛化占位标签
- 关键区域职责清晰，不彼此混淆
- 中间视图与对象类型及流程形态匹配
- 过滤器被正确拆分为页面级过滤器与主列本地过滤器
- 过滤器保持轻量，并服务工作，而不是压过工作本身
- 右侧抽屉支持上下文内处理
- 整体设计明显摆脱“过滤器 + 表格 + 行操作”的管理后台模式
