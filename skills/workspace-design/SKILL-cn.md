# Workspace Design

## 目标

把页面设计成工作空间：先理解当前局势，再收窄到真实工作，再直接处理真实对象。不要让页面退化成管理后台模式，即用户先筛记录、再找对象、最后才行动。

始终遵守这个顺序：

1. 先建立当前局势
2. 再收窄到正确的工作池
3. 再尽快落到真实对象
4. 再在最合适的视图里推进动作

规则冲突时，按以下优先级处理：

1. 先保证输出结构完整
2. 再满足硬规则
3. 在相关时保留 workspace 的层级关系
4. 再把通用标签改成业务语言
5. 最后才补充解释或示例

## 先解析输入

开始输出前，先抽取这些信息：

- 角色
- 主要目标
- 高频动作
- 风险或异常
- 核心对象
- 工作节奏
- 关键阶段或交接点
- 这项工作更偏队列驱动、分类驱动，还是混合模式
- 这次任务是否真的涉及移动端下钻、响应式、路由或浏览器历史
- 如果有现状，当前页面的问题是什么

如果信息不足：

- 做 1 到 5 条最小必要假设
- 放到 `0. 假设`
- 不要退回泛泛原则

## 选择模式

### 生成模式

当用户给的是需求、PRD、流程、角色目标或口头描述时，直接生成工作空间方案。

### 评审模式

当用户给的是截图、线框、HTML、现有页面方案，或者明确要求判断“像不像工作空间”时：

1. 先判断当前设计更像工作空间还是管理后台
2. 再指出结构性问题
3. 再给出完整的 workspace 重构方案

不要停在批评，必须产出具体重构结果。

## 默认结构模式

除非任务明显需要其他布局，否则优先参考这套结构：

```text
页面标题 + 一句话定位 + 全局快捷动作
势态卡横条
可选页面级过滤器 / scope 控件

双列 workspace：
左：第 1 层 ALL + 工作分类 或 第 2 层 Back + 任务队列
主：分类工作视图 或 Back + 任务详情 / 操作面
可选右侧抽屉
```

### 各区域的含义

- 顶部先建立局势，不先上大表格
- 左栏是分层工作查找器，不是应用导航
- 左栏应尽快从分类 scope 进入具体任务
- 中间主列才是真正的工作面
- 右侧抽屉可选，只承载次级上下文

## 响应式、移动端与路由

只有当任务真的涉及响应式、移动端流程、路由、浏览器历史，或依赖导航结构的实现细节时，才使用这一节。

### 桌面端

优先使用双列 workspace：

- 左：分层查找器
- 主：工作面
- 右侧抽屉：可选

### 移动端

不要强行保留并排双栏。

把同一套层级关系序列化成下钻流程：

```text
分类 -> 任务列表 -> 任务 -> 子任务 / 操作
```

### 路由模型

把 workspace 看成一个可导航的层级：

```text
workspace -> scope -> task -> step
```

这个层级怎么表达，取决于平台和应用约定：

- Path segments
- Query state
- 应用内导航状态
- Native stack navigation

只有当路由属于这次任务的一部分时，浏览器返回或应用内返回才需要遵守这个层级。

## 硬规则

无论生成还是评审，都应用这些规则：

1. 不要从表格开始设计页面
2. 不要用 `Manage X` 这类标题
3. 不要让过滤器成为页面主角
4. 不要把汇总卡做成没有动作价值的装饰
5. 不要默认所有对象都适合列表
6. 不要在桌面端强迫用户频繁跳页处理详情
7. 页面一进入就要回答“现在发生了什么，我该先做什么”
8. 左侧不是应用导航，而是分阶段工作查找器，并且应该尽快落到真实任务队列
9. 队列入口不是过滤器
10. 主工作区必须能直接推进核心动作
11. 主工作区里的列表是允许的，但前提是它是子任务列表、子对象列表，或当前任务内部的工作列表
12. 抽屉是可选的，只承载辅助上下文、次级动作和补充信息
13. 页面名、队列名、卡片名优先用业务动作语言和业务对象语言
14. 如果有队列分组，应来自流程、风险、时间压力或交接逻辑，而不是通用占位标签
15. 左侧无论一层还是两层，都要尽快落到可操作对象，而不是停在纯分类页
16. 如果左侧被理解成纯导航，说明设计正在偏离 workspace
17. 不要让左右两列都退化成嵌套的泛化列表；主列至少有一种状态必须是真正的工作面
18. 如果任务涉及移动端行为，应改成下钻流程，而不是把桌面双栏硬塞到窄屏
19. 如果任务涉及路由或浏览器历史，应保留有意义的导航状态，比如 scope、task 和主要操作步骤
20. 如果用了路由，像抽屉开关、分栏宽度、局部切换这类 incidental UI state，不应进 URL，除非任务明确要求

## 设计规则

### 1. 页面定位

优先提供：

- 页面标题
- 一句话定位

要求：

- 标题描述动作，不描述模块
- 标题优先用动词开头
- 定位语说明谁在使用、要达成什么目标、需要快速响应哪些风险或异常

### 2. 势态卡

势态卡用于建立局势，不是装饰性总量。

要求：

- 通常设计 4 到 6 张
- 每张卡都能下钻到真实工作
- 每张卡至少体现风险、趋势、时效中的一种

每张卡都说明：

- 名称
- 为什么重要
- 会下钻到哪个 scope、队列、视图或结果集
- 它对应哪个运营周期

适合放：

- 今日到期工作
- 临近超时项
- 高风险对象
- 当前瓶颈
- 本周完成率
- 容量压力
- 即将到期项

避免：

- 没有动作价值的静态总量
- 与本页职责无关的全局 KPI
- 纯 BI 风格统计

### 3. 左列：分层工作查找器

左列不是普通侧边栏导航，而是分阶段工作查找器。

当用户需要先选工作范围、再选具体队列时，优先使用两层结构。

#### 两层模式

```text
第 1 层：ALL + 工作分类
第 2 层：Back + 任务队列
```

以下情况默认更适合两层：

- 用户需要先选工作池
- 不同工作池有不同节奏
- 任务队列不能和应用导航混淆
- 主工作区需要更大宽度来做真实操作

#### 一层模式

```text
只有任务队列
```

以下情况更适合一层：

- 用户主要围绕一个对象池工作
- 分类切换不频繁
- 优先级可以靠排序、状态或时间压力表达

#### 要求

- 尽快落到具体任务队列
- 分类只是入口，不是最终停留页
- 如果用了两层左列，第 1 层要足够短，并明显是过渡层
- 任务队列必须包含真实工作对象，而不是伪装成队列的问题分类
- 队列项要能回答“我现在具体可以处理什么”

队列入口优先从这些维度推导：

- 流程阶段
- 风险类型
- 时间窗口
- 审批或交接步骤
- 恢复或升级路径
- 责任归属

只有在确实缺少更具体语义时，才使用：

- 待处理
- 风险
- 待跟进

对每个左列入口，都说明：

- 为什么存在
- 会进入什么工作
- 进入后任务队列长什么样
- 选中任务后主工作区进入什么状态

任务队列项通常包含：

- 对象标题
- 当前状态
- 时间压力
- 下一动作

### 4. 过滤器

过滤器不是必选项。根据场景，可以没有过滤器、只有一级过滤器，或者有两级过滤器。核心问题不是层级数量，而是过滤器是否真的帮助推进工作。

优先这样判断：

- 如果系统已经能明确推送相关任务，且页面目标很集中，可以不用过滤器
- 如果只需要改变整页 scope，用一级过滤器
- 如果既要控制整页 scope，又要独立控制主区呈现方式，再用两级过滤器

页面级过滤器定义整页上下文，只放那些真正会改变可用工作池的条件，例如：

- 时间
- 组织、区域、校区、项目
- 负责人
- 当前页面搜索

规则：

- 会改变左侧任务池，就放这里
- 通常显示 0 到 4 个
- 少而硬

主列本地过滤器只改变主工作区，例如：

- 视图切换
- 排序
- 分组
- 只看高风险
- 只看未完成

规则：

- 只改变主工作区
- 不要偷偷重建左侧任务池
- 可以更细，但不能喧宾夺主
- 如果主工作区已经足够清晰，可以省略

判断法：

- 会改变左侧任务池 -> 页面级过滤器
- 只改变主区视图 -> 本地过滤器
- 两者都不需要 -> 不放过滤器

不要把业务任务伪装成顶部过滤器。像 `待处理`、`逾期未完成`、`高风险` 这类标签，通常更适合放进左侧工作查找器或队列。

### 5. 主工作区

主列才是真正的工作面。

在双列模式里，主列通常有两种主要状态：

#### 状态 A：分类工作视图

进入某个分类后，主工作区可以显示分类级工作视图，例如：

- 卡片
- 列表
- 看板
- 时间线
- 日历
- 混合视图

当选中的工作池本身就适合一个结构化操作视图时，这种状态是合理的。

#### 状态 B：任务详情 / 操作面

选中某个任务后，主工作区应变成面向任务的操作面，例如：

- 详情视图
- 表单流
- 混合视图
- 带直接动作的卡片
- 子对象或子任务列表

先判断用户是在推进单个对象，还是在筛查一批对象。

如果是推进单个对象，优先考虑：

- 详情视图
- 表单流
- 卡片
- 混合视图
- 必要时使用子对象或子任务列表

如果是筛查一批对象，优先考虑：

- 列表
- 看板
- 卡片墙
- 图表加风险列表
- 当时间结构重要时，使用时间线或日历

以下场景通常不应默认成普通列表：

- 班次与人力安排
- 时间排程
- 现场执行
- 单对象处理
- 风险判断

需要明确：

- 默认视图
- 可切换视图
- 焦点对象或焦点任务
- 页面内可直接完成的核心动作
- 页面当前处于“分类工作视图”还是“任务操作面”状态

### 6. 主工作面里的列表

如果列表有助于完成当前选中任务，那么它在主工作面里是合理的。

适合的用法：

- 子任务
- 子记录
- checklist 步骤
- 当前 case 内的异常项
- 必须作为当前任务一部分处理的相关人员、课程或合同条目

避免把主区又做成一个泛化队列，除非这个工作本身就是批量筛查。

### 7. 右侧抽屉

抽屉是辅助区，不是主流程，而且它是可选的。

适合放：

- 规则说明
- 历史记录
- 备注
- 关联对象
- 候选项
- 次级动作
- 风险解释

不适合放：

- 整个主流程
- 一整套复杂子系统
- 本该由主工作区承载的核心处理动作

说明：

- 是否真的需要抽屉
- 什么时候打开
- 里面放什么
- 可以在里面完成哪些次级动作
- 为什么这些内容适合放在抽屉而不是主区

### 8. 宽度、分栏与响应式

只有当任务需要布局实现细节或响应式规则时，才使用这一节。

优先使用可由用户调整的双列桌面布局。

默认建议：

- 左列约 25% 到 30%
- 主列约 70% 到 75%
- 一个稳妥的起点是 30 / 70

实现建议：

- 优先用最小 / 最大宽度，而不是死百分比
- 左列在桌面端常见可用宽度约 320px 到 380px
- 主区应保持主导地位

分栏器建议：

- 桌面端允许用户调整
- 用 min/max 做边界
- 在可能时记住用户偏好
- 不要把分栏位置放进浏览器历史

响应式建议：

- 桌面端：双列布局
- 窄桌面 / 平板：保持主区主导，必要时缩小或折叠左列
- 移动端：把同一层级结构序列化成下钻流程

### 9. 路由与浏览器历史

只有当任务需要路由、深链接、浏览器历史或可分享导航状态时，才使用这一节。

把 workspace 看成一个可导航层级：

```text
workspace -> scope -> task -> step
```

路由规则：

- `workspace` = workspace 根
- `scope` = 当前选中的分类或工作池
- `task` = 当前选中的工作对象
- `step` = 主要操作步骤或子任务流程

返回行为通常应是：

```text
step -> task -> scope -> workspace
```

以下情况应 push history：

- 进入某个 scope
- 打开某个任务
- 进入某个主要操作步骤
- 切换到另一个任务或 scope

以下情况更适合 replace 或本地状态：

- 修正无效路由
- 恢复默认子视图
- 打开或关闭可选抽屉
- 改变分栏宽度
- 改变局部显示控件

深链接规则：

- 当路由属于任务范围时，链接应该可以直接打开
- 无效 scope 回退到 workspace root
- 无效 task 回退到 scope
- 无效 step 回退到 task
- 纠错回退优先用 replace

视图模式规则：

- 像 `cards`、`list`、`board`、`timeline`、`calendar` 这类会明显改变页面的 major view mode，才值得进入可分享状态
- 细碎 UI toggle 保持本地

## 反模式

如果以下现象占主导，页面就更像管理后台或伪 workspace：

- 一进页面先看到大表格
- 顶部堆满过滤器
- 左侧是模块树或应用导航栏
- 左侧只有问题分类，主区只是该分类下的任务列表
- 主区只是放大版列表加行尾动作
- 大多数动作都挂在行尾按钮
- 卡片只显示数字，不连接真实工作
- 抽屉只是详情垃圾桶
- 系统从不主动推送相关工作，用户必须先搜索
- 移动端试图保留桌面双栏，而不是改成下钻
- 浏览器返回破坏工作层级，或莫名退出页面

## 输出协议

默认按这个结构组织输出；如果调整顺序能让结果更好，可以合并或重排。

### 0. 假设

只在信息不足时输出。

### 1. 页面定位

- 页面标题
- 一句话定位

### 2. 势态卡

通常列 4 到 6 张。每张卡包括：

- 名称
- 为什么会改变优先级
- 会下钻到哪里
- 它代表哪个运营周期

### 3. 左列结构

- 是一层还是两层
- 如果是两层，定义：
  - 第 1 层：ALL + 工作分类
  - 第 2 层：Back + 任务队列
- 任务队列项模式

对每个入口都说明：

- 为什么存在
- 会进入什么工作
- 任务队列长什么样
- 选中任务后主区进入什么状态

### 4. 过滤器与快捷动作

- 是否需要过滤器
- 页面级过滤器
- 主列本地过滤器
- 高级过滤器
- 快捷动作

还要说明：

- 默认上下文如何建立
- 为什么选择无过滤器、一级过滤器或两级过滤器
- 默认展示哪些页面级过滤器（如果有）
- 默认展示哪些本地过滤器（如果有）

### 5. 主工作区

- 分类工作视图
- 任务详情 / 操作面
- 默认视图
- 可切换视图
- 焦点对象或焦点任务
- 页面内可直接完成的动作
- 主区里的列表是子任务列表、子对象列表，还是批量工作视图
- 为什么这个视图最适合这类工作

### 6. 右侧抽屉

- 是否需要
- 何时打开
- 包含什么
- 可以在里面完成哪些动作
- 为什么适合放在这里

### 7. 移动端下钻（按需）

只有当任务涉及响应式、移动端流程或明确的移动端要求时，才输出这一节。

描述移动端顺序，通常是：

- 分类
- 任务列表
- 任务
- 子任务 / 操作

说明：

- 哪些层级会合并
- 哪些动作留在任务页
- 哪些动作值得再下钻一层

### 8. 路由与历史（按需）

只有当任务涉及路由、浏览器历史、深链接或可分享导航状态时，才输出这一节。

说明：

- 路由层级：workspace -> scope -> task -> step
- 哪些状态应进入可分享导航状态
- 哪些状态保持本地
- 返回行为
- 重要 query 或 view state

## 评审模式附加输出

在评审模式下，把这些放在 `1. 页面定位` 前面：

### 当前判断

- 当前设计更像：工作空间 / 管理后台 / 混合但正在漂移

### 主要问题

- 通常列 3 到 8 条具体问题
- 指向结构、节奏、动作承接、队列语义、视图选择、过滤器层级，以及在相关时的移动端或路由行为
- 可以指出左侧被理解成导航，而不是工作查找器
- 也可以指出页面根本不需要过滤器却硬塞了过滤器
- 不要只写抽象批评

然后给出完整重构方案，覆盖上述核心结构。

## 语言要求

优先使用：

- 业务动作语言
- 业务对象语言
- 时间压力语言
- 风险语言
- 听起来像“工作”的队列语言，而不是模块语言

避免使用：

- 字段语言
- 数据库语言
- 泛化系统标签
- 明明表达的是工作 scope，却写成应用导航语言

优先写：

- 待审核
- 已阻塞
- 7 天内到期
- 需要补资料
- 可发布
- 考勤异常
- 续费动作
- 待确认合同

避免写：

- 状态 = 待处理
- 优先级 = 高
- 类型 = 常规
- 合同管理
- 学生模块

## 完成标准

只有同时满足以下条件，才算完成：

- 页面名称描述动作，而不是模块
- 顶部先建立局势，并连接到真实工作
- 左侧扮演分阶段工作查找器，而不是应用导航
- 左侧结构合理地选择了一层或两层，并且能快速落到可操作对象
- 队列入口来自流程、风险、时间压力或交接逻辑
- 过滤器策略合理，允许无过滤器、一级过滤器或两级过滤器
- 主工作区能直接推进核心动作
- 主区内的列表有正当理由，比如批量工作视图、子任务列表或子对象工作列表
- 抽屉保持辅助且可选
- 当任务涉及移动端行为时，移动端用下钻而不是硬塞桌面双栏
- 当任务涉及路由或浏览器历史时，它们支持 workspace -> scope -> task -> step 的层级
- 页面明显摆脱“过滤器 + 表格 + 行操作”的管理后台模式
- 输出覆盖关键页面结构与设计决策，并且只在相关时覆盖移动端或路由行为
